# Proposal 0001: Personal Compute Adapter

## Status

Draft.

## Problem

SpecGraph and SpecPM both need compute-heavy repository understanding:

- repository checkout and file inventory
- deterministic language and symbol extraction
- LLM-backed intent summarization
- capability extraction
- graph or package assembly
- validation and provenance capture

If the central service performs all of this work, it becomes a crawler, LLM
inference farm, CI-like processor, registry, billing sink, and trust authority
at the same time.

SpecNode separates local compute from central orchestration.

## Decision

Introduce `specnode`: a local agent that runs on a user's device, opens an
outbound connection to SpecGraph or SpecPM, accepts typed jobs, executes them in
a bounded workspace, calls local OpenAI-compatible providers, and returns
structured artifacts.

The control plane provides:

- pairing
- node registry
- job queue
- policy distribution
- artifact validation
- registry/search/index updates
- audit and observability

The node provides:

- local provider access
- local repository access
- deterministic analysis
- LLM inference
- artifact hashing
- usage receipts
- optional signing

## Non-Goals

- No public compute marketplace.
- No payment layer.
- No untrusted third-party worker execution.
- No inbound network exposure on the user's device.
- No cloud access to local LLM endpoints.
- No arbitrary prompt or shell execution API.
- No full Agent Passport runtime requirement for the first MVP.

## Architecture

```text
SpecGraph / SpecPM Web UI
        |
        v
Node Control Plane
  - pairing
  - node registry
  - job queue
  - policy
  - audit log
        |
        | WSS jobs
        v
specnode
  - job executor
  - repo workspace
  - local LLM provider adapter
  - artifact validator
  - usage receipt writer
        |
        | localhost
        v
Ollama / LM Studio / llama.cpp / custom OpenAI-compatible endpoint
```

## SpecGraph Mode

SpecGraph can use SpecNode for bounded extraction tasks:

- `specgraph.extract_file_summary`
- `specgraph.extract_module_intent`
- `specgraph.extract_capabilities`
- `specgraph.build_graph_fragment`

Early mode:

```text
cloud chunks input -> node performs local inference -> node returns structure
```

Later mode:

```text
cloud sends repo URL + commit + task plan -> node processes locally -> node returns artifacts
```

The second mode is better for private repositories and large projects because
source content does not need to pass through the cloud.

## SpecPM Mode

SpecPM should prefer maintainer-side package generation:

```bash
specnode specpm init
specnode specpm build --repo .
specnode specpm validate
specnode specpm publish
```

The registry should receive normalized packages:

- `spec-package.yaml`
- `specgraph.json`
- `provenance.json`
- conformance hints or tests
- artifact hashes
- optional signature

This lets SpecPM operate as a registry, validator, resolver, and search index
instead of a centralized repository-processing factory.

## Job Protocol Requirements

Jobs must include:

- stable job id
- service owner: `specgraph` or `specpm`
- typed job name
- input references
- explicit policy
- output schema
- timeout and resource limits

Jobs must not include:

- unrestricted prompts
- shell scripts generated by the service
- requests to read outside the workspace
- requests to expose local provider endpoints

## Security Requirements

- Repository content is untrusted.
- LLM output is untrusted.
- The local workspace is the filesystem boundary.
- Network access is deny-by-default and job-scoped.
- Shell access is deny-by-default.
- LLM tools are disabled in the MVP.
- Output must pass schema validation before upload.
- Artifacts should be hash-linked to provenance.

## Validation Criteria

The proposal is exercised when:

- a node can pair through an outbound channel
- provider discovery reports at least one local model
- a typed SpecGraph job returns schema-valid output
- a typed SpecPM package job returns a candidate package
- both jobs produce usage receipts
- the control plane can reject malformed artifacts

## Open Questions

- Which language should the first runtime use?
- Should pairing be account-bound, project-bound, or service-bound?
- What is the first required local provider: Ollama only, or generic
  OpenAI-compatible first?
- Which artifact signatures are required before public SpecPM submissions?
- How much source evidence should the node upload for registry review?
